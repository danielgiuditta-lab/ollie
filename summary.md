# Spaces Platform — Technical & Product Architecture

Welcome to the **Spaces Platform**, a state-of-the-art, AI-powered developer environment designed to enable seamless, natural-language frontend prototyping directly within Google Drive. By combining standard Google Workspace OAuth authentication with the advanced capabilities of the Google GenAI Interactions SDK, this application converts text descriptions into fully interactive, client-side web applications with dynamic Google Drive context ingestion and round-trip state saving.

---

## 1. Product Vision & UX Design

The **Spaces Platform** leverages a professional, polished and clean visual aesthetic modeled after a modern Google Drive extension. It serves as a visual staging playground where non-technical and technical users can collaboratively build individual applications, utilities, and documentation through a streamlined conversational chat panel.

### Core Visual Philosophy
* **Modern Google Drive Aesthetic**: High-contrast light canvas layout featuring gentle off-white surfaces (`#F8FAFD`), customized subtle glowing background gradients, spacious rounded containers (`rounded-[32px]`), and responsive card transitions.
* **Typographic Rigor**: Standardized pairing of legible UI typography with specialized code/metrics representation (e.g. JetBrains Mono for status counters) to enforce visual rhythm.
* **Proportional Craft**: Clean layouts stripped of low-quality "AI slop" or cluttering telemetry logs, relying entirely on beautiful empty state illustrations, unified Google Drive file icon pairings, and interactive progress loaders.

### Core Visual staging & Header Architecture
* **Single Unified Header**: The app features a single top-level `CanvasHeader` component that replaces split document bars. It renders unclosable space names, breadcrumbs, suggested member avatars, Native Google Drive document open triggers, and collapsible side panel toggles.
* **Original Design Philosophy**: Maintains the original canvas styling—omitting bottom border separators (hair lines), capsule background highlights on sub-item/artifact breadcrumbs, and extra margin configurations. Sub-items are rendered with standard text styling sized to match the space parent item.
* **The Collapsible Right Sources Panel**: The file navigation sidebar (`CanvasSidebar`) is positioned on the right of the canvas viewport, using a single-column layout with no headers/titles. Clicking the folder icon in the header toggles the panel, and clicking the close button (`X`) within the panel collapses it.

### The 4-State Canvas Architecture
The main visual stage is a responsive container that routes between distinct state variations depending on user flows and workspace context:
1. **The Dashboard (To-Do & Agenda View)**:
   - Serves as the primary landing page upon user login and authentication.
   - Summarizes the synthesized action plan retrieved from the user's Workspace activities logs.
   - Renders flat, stacked `InferredTaskCard` cells displaying pending loader animations, metadata Badges, and proactive draft previews.
2. **The Null State (Workspace Ingestion)**:
   - Displays when a user is first establishing their context or has not selected a workspace.
   - Renders a custom sign-in component using Google Drive OAuth.
   - Lists the user's **Recent Files** modified in Drive via direct fetch APIs, allowing them to click a file or folder to "ingest" it into the assistant's sandbox as starting context.
2. **The App View (Live Preview)**:
   - Runs the generated application inside a highly secure sandbox iframe equipped with strict sandbox permissions (`allow-scripts allow-forms allow-popups allow-same-origin allow-modals allow-downloads`).
   - Automatically inlines referenced CSS and JavaScript modules compiled by the engine.
   - Continuously monitors computed styles (e.g., using a short-polling `getComputedStyle` reader in the iframe body) to detect the generated app's theme brightness (light vs. dark) and align the outer studio canvas wrapper's background seamlessly.
3. **The Files List (Directory View)**:
   - Presents a flat directory structure of all code modules generated by the agent or ingested from Drive.
   - Maps file attributes directly into an elegant table detailing owner profiles, sizing, and modification dates.
4. **The File Detail (Split View / File Viewer)**:
   - Activates when the user selects a code file from the directory.
   - Splits the layout: showcases a compact navigation index on the left, alongside an interactive syntax panel on the right rendering stringified content or Markdown documents with beautiful styling.
   - Features a **Save to Drive** action, safely syncing changes back into Google Drive using hot-swap PATCH endpoints.

## 2. Dynamic Data Core & Integrations

The workspace connects directly to real APIs to eliminate mock placeholders, guaranteeing authentic, round-trip, secure data interactions.

```
       ┌────────────────────────┐
       │   Google Drive API     │
       └─────┬────────────▲─────┘
             │            │
      Ingest │            │ Save
      Context│            │ File
             ▼            │
  ┌─────────────────────────────┐
  │     Express App Server      │
  │     (server.ts middleware)  │
  └──────────┬────────────▲─────┘
             │            │
  SSE Stream │            │ Ingestion Context
  vibe-code  ▼            │ & User Prompts
  ┌─────────────────────────────┐
  │     Spaces Browser      │
  │  (React 19 / Vite Frontend) │
  └─────────────────────────────┘
```

### Authentication & API Security
* **Google OAuth2 Gateway**: Instantiates authentication via `@react-oauth/google`, specifically requesting OAuth scopes for `profile, email` and full Google Drive file control (`https://www.googleapis.com/auth/drive`).
* **Zero Client-Side Secrets**: All interaction logic involving LLMs and backend sandbox controllers flows securely through server proxy routes, keeping API keys protected.

### Context Ingestion Pipeline (`/api/ingest-context`)
When a user clicks a file or folder in the primary Drive table:
1. The client relays the `folderId` along with the user's OAuth access token via an `Authorization` bearer header.
2. The server-side API queries Google Drive (`https://www.googleapis.com/drive/v3/files/${folderId}`) to detect the resource type:
   * **If a file**: It downloads the file contents as raw text using Google Alt-Media media download protocols (`?alt=media`).
   * **If a folder**: It lists matching child hierarchies (`q='${folderId}'+in+parents`) excluding internal configurations, maps their definitions, and concurrently downloads all available files within that context.
3. The server responds with `{ files: [...] }` containing raw text schemas, which the frontend ingests directly to configure the active workspace instance.

## 3. The "Two Brains" Paradigm & Core Generation Engine

When assessing the AI intelligence powering this environment, it is critical to distinguish between two completely independent AI entities operating in parallel:

```
┌───────────────────────────────────────┐         ┌────────────────────────────────────────┐
│          BUILDER AGENT                │         │             IN-APP AGENT               │
│         (Google AI Studio)            │         │      (Antigravity Sandbox Server)      │
├───────────────────────────────────────┤         ├────────────────────────────────────────┤
│ • Responsible for building this      │         │ • Governed by 'agent-system-prompt.md' │
│   React + Express codebase.           │   vs    │ • Interacts with the end-user in chat. │
│ • Compiles layout, styling, streams.  │         │ • Lives in a remote Linux Sandbox.     │
│ • Never alters the running app itself.│         │ • Generates code widgets dynamically.  │
└───────────────────────────────────────┘         └────────────────────────────────────────┘
```

### The In-App Agent (Antigravity Sandbox)
* **Under the Hood**: Uses the official Google GenAI Interactions SDK to spin up a managed remote Linux container running `antigravity-preview-05-2026`.
* **System Prompt Constraints**: The behavior of the In-App Agent is strictly governed on the server side. During the `/api/vibe-code` post request, the server reads the local markdown file `/src/agent-system-prompt.md` and passes it as the `system_instruction` to `ai.interactions.create()`.
* **Environment State Persistence**: Rather than initiating empty stateless prompts, the server returning the SSE streams extracts the `environment_id` returned on the final interaction completed hook. This ID is saved on the client as `envId`. For subsequent user chats, the React app forwards the exact same ID back to `/api/vibe-code`, preserving the state of the remote Linux filesystem.

### Problem-Driven Intent Router (`/api/classify-intent`)
* **Structured Intent Classification**: Before executing code stream or document generation, unhandled chat messages hit `/api/classify-intent` running `gemini-3.5-flash` with JSON schema enforcement.
* **Document Request Guard**: Requests to write PRDs, specifications, roadmaps, or articles route directly to **Doc Journey** (`/api/doc-journey`), generating native Google Docs (`.doc`) / Google Slides (`.gslides`) in paper card layout. *No app is vibecoded.*
* **Tailored Tool Archetype Proposals**: Workflow problems (e.g. *"help me track the team's work"*, *"manage software bugs"*, *"track decisions & risks"*) infer optimal specialized archetypes (`kanban`, `bug_tracker`, `decision_risk_log`, `approval_queue`) and present an interactive proposal pill (e.g. `[⚡ Build Kanban Board]`) in the bot message bubble before executing code stream.

## 4. Sandbox Execution vs. Client-Side Parsing (Current State & Analysis)

### What We Have (Currently Built)
* **Continuous Streaming Feedback**: Real-time event streams (`text/event-stream`) map server-sent chunks directly into animated visual step indicators in the chat sidebar.
* **Sandbox Association & Mandated Environment Parameter**: A persistent `env_id` is successfully preserved and relayed. For fresh sandboxes or defaults, `environment: "remote"` is explicitly and mandatorily passed to `ai.interactions.create()` in `server.ts` (`const resolvedEnv = (env_id && env_id !== 'remote' && env_id !== 'null' && env_id !== 'undefined') ? env_id : "remote"`), preventing `HTTP 400 Missing required field 'environment'` API rejections.
* **Single Loading State & Seamless Canvas Mount**: When initiating vibe-code generation, the main canvas viewport remains cleanly on the Space Dashboard or Home view while the left chat sidebar renders building progress steps. `<AppView/>` dynamically transitions `viewState` to `'app'` mode the exact instant live HTML content (>30 chars) is extracted or when `interaction.completed` fires, eliminating duplicate loading spinners and blank screens.
* **Universal Stream Extraction & Event Delta Parsing**: Client-side stream listeners dynamically resolve event types (`eventType = event.event_type || event.type || event.event || event.kind || ...`) and extract text deltas across root-level `.text`, `.code`, `.content`, and `.delta` payload objects, ensuring text chunks reliably accumulate into `accumulatedOutput` and render live interactive HTML.
* **Client-side App Preview Compiler**: Because the client compiles components on the fly instead of pointing to a remote web server, `<AppView/>` retrieves the extracted `index.html` file and dynamically inlines referenced local JavaScript and CSS resources via regex replace methods before feeding it directly into a highly restricted sandbox iframe's `srcDoc` attribute.

### What We Do Not Have (Missing / Slated for Future Upgrade)
* **No Client-to-Sandbox Directory Querying**: The local file navigation views do *not* actually request a structured filesystem manifest directly from the remote sandbox to render the tree. It depends entirely on parsing the conversational stream's output. If the agent changes a file in the remote container but omits printing it in full markdown inside the subsequent turn, the client-side list falls out of sync.
* **No Direct Host Routing Preview**: The `AppView` does not connect over a live port forwarding to an active web server hosted inside the container sandbox. Dynamic interactions inside the generated widgets are forced back to static client-side single-page scopes.

### Sandboxed Execution Limits (Exit Code 137 / SIGKILL)
* **The Problem**: Running in development mode via `npm run dev` spawns `tsx` (TypeScript execute) and an in-memory Vite dev server inside Express. On-the-fly compilation of TSX and assets uses considerable memory. In resource-restricted VM containers or sandboxes, this exceeds RAM allocations, leading the environment to kill the process with `SIGKILL` (exit code `137`), rendering the preview unreachable.
* **The Fix**: Pre-compile all client/server assets by running `npm run build` (which builds static frontend bundles in `/dist` and uses `esbuild` to compile `server.ts` to `dist/server.cjs`). Then run the pre-compiled server in production mode using `npm run start`. Bypassing runtime compilers lowers memory usage to <50MB RAM, allowing it to run stably.

## 5. Security & Write Boundaries

### Authentic OAuth Restrictions
* **Selected Scope Hardening**: In congruence with Google Drive security requirements, write-actions are bounded. The app can read user files, but patch syncing is restricted strictly to synchronizing code outputs of loaded contexts.
* **Flat-File Database Security Constraint**: To prevent unauthorized leakage of OAuth tokens or high-privilege access, the Antigravity agent operating inside the remote Linux sandbox is strictly forbidden from attempting to authenticate or write directly to native Google Drive assets (like trying to spin up a live Google Sheet write action inside the user's account). Instead, any stateful interaction generated by the In-App helper MUST rely exclusively on flat storage files (like `.json` or `.csv`) located securely inside the sandboxed filesystem container.

## 6. Future Outlook / Developer Roadmap

Features marked as entirely absent or partially decoupled from current implementations represent key priorities for future enhancements:

1. **True Sandbox Port-Forwarding (Web Preview hosting)**
   * **Scope**: Transition the visual stage preview iframe from a regex-parsed client-side `srcDoc` into a real, live-hosted web preview server running directly inside the container.
   * **Mechanism**: The Antigravity Sandbox spins up a lightweight node or python server natively, allowing `<AppView/>` to route actions via standard sandbox URLs.
2. **Structured File Manifest System**
   * **Scope**: Replace client-side markdown stream parsing with direct container mapping.
   * **Mechanism**: Query the sandboxed file tree periodically using a container-side runner (or structured json report files written by the agent) to guarantee file trees in `<FilesList/>` and `<FileViewer/>` perfectly mirror the sandbox disk layout.
